## 一、通用代码统一到 `common/` 目录

### 1.1 目标结构

将 `types`、`utils`、`constants`、`composables`、`components` 统一收入 `src/common/`：

```
src/
├── main.ts
├── App.vue
├── common/                          # ← 所有与业务无关的通用代码
│   ├── components/                  # 通用 UI 组件
│   │   └── FileUpload.vue
│   ├── composables/                 # 组合式函数（可复用逻辑）
│   │   ├── useFilePreview.ts
│   │   └── useAuth.ts
│   ├── constants/                   # 常量
│   │   └── storage.ts
│   ├── types/                       # 全局类型定义
│   │   └── index.ts
│   └── utils/                       # 工具函数
│       ├── http.ts
│       └── logger.ts
├── api/                             # API 层（按业务域分模块）
│   ├── types.ts
│   └── modules/
├── assets/
├── router/
├── stores/
└── views/
```

### 1.2 路径别名配置

在 `vite.config.ts` 和 `tsconfig.app.json` 中添加 `@common` 别名，方便导入：

**vite.config.ts：**
```typescript
resolve: {
  alias: {
    '@': fileURLToPath(new URL('./src', import.meta.url)),
    '@common': fileURLToPath(new URL('./src/common', import.meta.url)),
  },
},
```

**tsconfig.app.json：**
```json
"paths": {
  "@/*": ["./src/*"],
  "@common/*": ["./src/common/*"]
}
```

### 1.3 使用效果

```typescript
// 优化前
import { useFilePreview } from '@/composables/useFilePreview'
import { STORAGE_KEYS } from '@/constants/storage'
import type { ApiResponse } from '@/types'

// 优化后 —— 来源一目了然，是通用代码而非业务代码
import { useFilePreview } from '@common/composables/useFilePreview'
import { STORAGE_KEYS } from '@common/constants/storage'
import type { ApiResponse } from '@common/types'
```

### 1.4 为什么 `api/` 不放入 `common/`

API 模块是**业务相关**的（每个项目的接口完全不同），不具备通用性。`common/` 应只放"换一个项目也能直接复用"的代码。判断标准：

| 放入 `common/` | 不放入 `common/` |
|---|---|
| `http.ts`（axios 封装） | `api/modules/score.ts`（加分业务） |
| `useFilePreview.ts`（文件预览逻辑） | `stores/user.ts`（用户状态） |
| `STORAGE_KEYS`（localStorage 常量） | `router/modules/`（业务路由） |
| `FileUpload.vue`（通用上传组件） | `views/`（业务页面） |

```
├── .prettierrc.json

├── tsconfig.json           ┐
├── tsconfig.app.json       ├ 可精简
├── tsconfig.node.json      ┘
├── tsconfig.vitest.json    ← 不写测试则删除
├── vite.config.ts
└── vitest.config.ts        ← 不写测试则删除
```

### 2.2 升级 Tailwind v4：删掉 3 个文件

**Tailwind CSS v4 是一次革命性变化**，直接消除 `tailwind.config.js`、`postcss.config.cjs` 以及 `autoprefixer` 依赖。

#### 变化对比

| 项目 | Tailwind v3（当前） | Tailwind v4 |
|------|---|---|
| 配置方式 | JavaScript（`tailwind.config.js`） | CSS（`@theme` 指令） |
| PostCSS 依赖 | 需要 `postcss.config.cjs` + `autoprefixer` | **不需要**，Vite 插件内置处理 |
| 构建引擎 | JavaScript | Rust（5 倍速度提升） |
| 内容扫描 | 手动配置 `content` 数组 | **零配置自动检测** |
| 暗色模式 | `darkMode: 'class'` in JS | `@custom-variant dark` in CSS |

#### 迁移步骤

**1. 替换依赖**

```bash
# 卸载旧依赖
pnpm remove tailwindcss postcss autoprefixer

# 安装新依赖
pnpm add tailwindcss@latest @tailwindcss/vite
```

**2. 删除不再需要的文件**

- ❌ 删除 `postcss.config.cjs`
- ❌ 删除 `tailwind.config.js`

**3. 修改 `vite.config.ts`**

```typescript
import tailwindcss from '@tailwindcss/vite'

export default defineConfig({
  plugins: [
    tailwindcss(),   // ← 直接作为 Vite 插件
    vue(),
    vueDevTools(),
  ],
})
```

**4. 修改 `src/assets/main.css`**

```css
/* 优化前（v3） */
@tailwind base;
@tailwind components;
@tailwind utilities;

/* 优化后（v4） */
@import "tailwindcss";

/* 暗色模式（替代 tailwind.config.js 中的 darkMode: 'class'） */
@custom-variant dark (&:is(.dark *));

/* 自定义主题（替代 tailwind.config.js 中的 theme.extend） */
@theme {
  --color-dark-blue: #102252;
}

/* 字体保持不变 */
@font-face {
  font-family: 'ZNtitle';
  src: url('@/assets/fonts/ZNtitle.ttf') format('truetype');
  font-weight: normal;
  font-style: normal;
}

@font-face {
  font-family: 'ENtitle';
  src: url('@/assets/fonts/ENtitle.ttf') format('truetype');
  font-weight: normal;
  font-style: normal;
}
```

你原来 `tailwind.config.js` 中的自定义颜色 `dark-blue: '#102252'`，现在用 `@theme { --color-dark-blue: #102252; }` 在 CSS 中直接声明，使用方式不变（`bg-dark-blue`、`text-dark-blue` 等）。

> Tailwind 官方提供了自动升级工具 `@tailwindcss/upgrade`，可自动处理约 90% 的迁移。

### 2.3 `.editorconfig` 分析：保留还是删除？

#### `.editorconfig` 和 ESLint/Prettier 的职责对比

| 规则 | `.editorconfig` | Prettier | ESLint |
|------|:---:|:---:|:---:|
| 缩进大小 | ✅ | ✅ | ❌ |
| 缩进风格（空格/Tab） | ✅ | ✅ | ❌ |
| 行尾换行（LF/CRLF） | ✅ | ✅ | ❌ |
| 文件末尾空行 | ✅ | ✅ | ❌ |
| 尾部空白 | ✅ | ✅ | ❌ |
| 字符编码 | ✅ | ❌ | ❌ |
| 最大行宽 | ✅ | ✅ | ❌ |

**结论：职责几乎完全重叠。**

- **如果团队统一使用 Prettier 格式化** → `.editorconfig` 可以删除
- **如果团队有人不装 Prettier 插件** → `.editorconfig` 作为兜底保留

**我的建议**：对于个人项目或小团队，Prettier 已经足够，**可以删除 `.editorconfig`**。如果做通用模板给其他人用，保留更稳妥（因为 `.editorconfig` 几乎所有编辑器原生支持，零配置）。

### 2.4 `env.d.ts` 精简

当前内容只有一行：

```typescript
/// <reference types="vite/client" />
```

建议扩展为声明环境变量类型（这是 `env.d.ts` 存在的核心价值）：

```typescript
/// <reference types="vite/client" />

interface ImportMetaEnv {
  readonly VITE_APP_TITLE: string
  readonly VITE_BASE_API: string
}

interface ImportMeta {
  readonly env: ImportMetaEnv
}
```

这样当你写 `import.meta.env.VITE_BASE_API` 时，TypeScript 会提供自动补全和类型检查。**这个文件不能删除**，但只需这几行即可。

### 2.5 tsconfig 精简分析

#### 当前状况：4 个 tsconfig 文件

```
tsconfig.json            → 项目引用入口（references）
tsconfig.app.json        → 浏览器端应用代码
tsconfig.node.json       → Node 端（Vite 配置文件等）
tsconfig.vitest.json     → 测试代码
```

#### 为什么需要多个 tsconfig？

TypeScript 的 `project references` 功能允许**针对不同运行环境设置不同的编译选项**。核心矛盾在于：

| 环境 | 可用 API | 示例 |
|------|---------|------|
| 浏览器（app） | DOM API（`document`、`window`） | `src/` 下的 Vue 组件 |
| Node.js（node） | Node API（`fs`、`path`、`process`） | `vite.config.ts` |
| 测试（vitest） | 两者都需要 + jsdom 模拟 | `__tests__/` |

如果只用一个 tsconfig，要么浏览器代码能调用 `fs`（不安全），要么 Vite 配置不能用 `process`（报错）。

#### 能否减少到 2 个？

**可以**。在你不写单元测试的情况下：

**`tsconfig.json`（主配置）：**
```json
{
  "files": [],
  "references": [
    { "path": "./tsconfig.app.json" },
    { "path": "./tsconfig.node.json" }
  ]
}
```

**`tsconfig.app.json`（应用代码）：**
```json
{
  "extends": "@vue/tsconfig/tsconfig.dom.json",
  "include": ["env.d.ts", "src/**/*", "src/**/*.vue"],
  "compilerOptions": {
    "tsBuildInfoFile": "./node_modules/.tmp/tsconfig.app.tsbuildinfo",
    "paths": {
      "@/*": ["./src/*"],
      "@common/*": ["./src/common/*"]
    }
  }
}
```

**`tsconfig.node.json`（构建工具）：**
```json
{
  "extends": "@tsconfig/node22/tsconfig.json",
  "include": ["vite.config.*"],
  "compilerOptions": {
    "noEmit": true,
    "tsBuildInfoFile": "./node_modules/.tmp/tsconfig.node.tsbuildinfo",
    "module": "ESNext",
    "moduleResolution": "Bundler",
    "types": ["node"]
  }
}
```

**变化**：
- 删除 `tsconfig.vitest.json`（不写测试）
- `tsconfig.node.json` 的 include 从 6 项清理到 1 项（只保留 `vite.config.*`）
- `tsconfig.json` 的 references 从 3 个减到 2 个

#### 能否只用 1 个 tsconfig？

**技术上可以但不推荐**。原因：

1. `vite.config.ts` 需要 Node 类型（`process`、`__dirname`），而你的 Vue 代码需要 DOM 类型（`document`、`window`）
2. 如果合并成一个，你的 Vue 组件里可以调用 `require('fs')` 而不报错 —— 类型检查失去了意义
3. Vue 官方脚手架 `create-vue`、Vite 官方模板、`@vue/tsconfig` 都基于多文件设计

**实际上 3 个文件的 `tsconfig` 方案在 Vue 生态是标准实践**，不是冗余。其他框架（如 Next.js）看起来只有一个 `tsconfig.json`，是因为框架内部帮你处理了这个分离。

## 三、配置文件能否放入统一目录？

### 3.1 结论：不能，但有替代方案

这是一个很常见的诉求 —— 根目录配置文件太多看着乱。但遗憾的是：

**Vite 团队在 2024 年明确拒绝了 `.config/` 目录的提案**（[vitejs/vite#15911](https://github.com/vitejs/vite/issues/15911)），理由：

1. **性能开销**：Vite 需要额外的文件系统查找来定位配置
2. **`tsconfig.json` 的路径语义**：TypeScript 使用 tsconfig 所在的目录作为基准路径，移动 tsconfig 会导致所有相对路径失效
3. **生态碎片化**：不同工具对配置目录的支持不一致，移动部分文件会更混乱

具体每个文件的限制：

| 文件 | 能否移入子目录 | 原因 |
|------|:---:|------|
| `vite.config.ts` | ❌ | Vite 硬编码读取项目根目录，无配置项指定路径 |
| `tsconfig*.json` | ❌ | TypeScript 以 tsconfig 所在目录为 `rootDir`，移动后所有路径引用断裂 |
| `package.json` | ❌ | npm/pnpm 要求在项目根目录 |
| `index.html` | ❌ | Vite 以此作为 SPA 入口，必须在根目录 |
| `.prettierrc.json` | ⚠️ 可以但不推荐 | 可通过 `package.json` 的 `prettier` 字段指定，但不值得 |
| `.gitignore` / `.gitattributes` | ❌ | Git 要求在仓库根目录 |
| `env.d.ts` | ⚠️ 技术上可以 | 但需修改 tsconfig 的 include，增加复杂度 |
| `.env.*` | ⚠️ 可以 | Vite 支持 `envDir` 选项指定环境文件目录 |

### 3.2 替代方案：VSCode 文件嵌套（推荐）

Vite 团队推荐的方案是利用 VSCode 的 **文件嵌套（File Nesting）** 功能，将关联的配置文件视觉上折叠到 `package.json` 下：

在 `.vscode/settings.json` 中添加：

```json
{
  "explorer.fileNesting.enabled": true,
  "explorer.fileNesting.expand": false,
  "explorer.fileNesting.patterns": {
    "package.json": "tsconfig*.json, vite.config.*, .prettierrc*, .editorconfig, .gitattributes, .gitignore, env.d.ts, .env*"
  }
}
```

效果：在 VSCode 资源管理器中，所有配置文件会折叠到 `package.json` 下面，展开才可见。根目录看起来非常干净：

```
▶ package.json          (点击展开可看到所有配置文件)
  index.html
▶ src/
▶ public/
```

### 3.3 `.env` 文件可以单独收入目录

虽然大部分配置不能移动，但 `.env` 文件是个例外。Vite 提供了 `envDir` 选项：

```typescript
// vite.config.ts
export default defineConfig({
  envDir: './config/env',
})
```

然后将 `.env.dev`、`.env.prod` 移入 `config/env/` 目录。但个人建议：**不值得**，因为只移动 2-3 个文件反而增加了寻找配置的认知成本。

### 3.4 最终建议

> **不移动任何配置文件，使用 VSCode 文件嵌套让根目录视觉整洁。**

精简配置文件数量本身（从 16 减到 11）已经是最有效的"让根目录更干净"的方式。

## 四、路由守卫深度分析

### 4.1 什么是路由守卫？

路由守卫（Navigation Guards）是 Vue Router 提供的**钩子函数**，在路由跳转的不同阶段执行，可以拦截、重定向或取消导航。

用最直白的比喻：**路由守卫就是你家门口的保安**。

- 有人要进入你家（访问某个页面）
- 保安先检查身份（有没有登录 token）
- 身份合格 → 放行（`next()`）
- 身份不合格 → 赶去登录页（`next('/login')`）
- 身份合格但信息不全 → 先去补充信息（`next('/profile')`）

### 4.2 路由守卫的三种类型

```
用户点击链接 / 输入 URL
        │
        ▼
┌─────────────────────────┐
│  全局前置守卫 beforeEach  │  ← 每次导航都会触发（最常用）
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│  路由独享守卫 beforeEnter │  ← 只在进入特定路由时触发
└────────────┬────────────┘
             ▼
┌─────────────────────────┐
│  组件内守卫              │  ← 在组件内部定义
│  onBeforeRouteLeave     │
│  onBeforeRouteUpdate    │
└────────────┬────────────┘
             ▼
        页面渲染
```

### 4.3 你的项目当前状况

```typescript
// router/index.ts - 当前代码
router.beforeEach(async (to, from, next) => {
  next();  // ← 直接放行，相当于保安在睡觉
})
```

这个守卫**完全没有起作用**。导入了 `useUserStore` 和 `ElMessage` 但都没用到。

#### 这导致的问题

现在你的项目中，认证逻辑是这样分散的：

```
用户直接访问 /home/score/index
        │
        ▼ 路由守卫（空壳，直接放行）
        │
        ▼ 页面加载，发起 API 请求
        │
        ▼ http.ts 响应拦截器检测到 401/403
        │
        ▼ clearTokensAndRedirect() → 跳转登录页
```

问题在于：
1. **页面会先渲染出来**，然后 API 请求失败，再跳走 —— 用户看到页面闪一下就消失了
2. **不需要 API 的页面**永远不会被拦截 —— 未登录用户可以看到页面骨架
3. **用户体验差** —— 每个 API 请求都要单独处理认证失败

### 4.4 路由守卫能解决什么问题

| 场景 | 没有守卫 | 有守卫 |
|------|---------|--------|
| 未登录访问 `/home` | 页面渲染 → API 报错 → 跳登录页（闪屏） | **直接拦截**，跳转登录页（无闪屏） |
| Token 存在但用户信息未加载 | 每个页面自己 `fetchUserInfo` | 守卫**统一加载一次**，所有页面直接用 |
| 已登录再访问 `/login` | 正常显示登录页（不合理） | **自动跳转**到首页 |
| 需要特定权限的页面 | 页面内部自己判断权限 | 守卫**统一拦截** |
| 页面跳转时显示加载条 | 无法实现 | 守卫中控制 NProgress 进度条 |

### 4.5 推荐的守卫实现

```typescript
// router/index.ts
import { createRouter, createWebHistory } from 'vue-router'
import { useUserStore } from '@/stores/user'
import { STORAGE_KEYS } from '@common/constants/storage'

// 不需要登录就能访问的页面（白名单）
const PUBLIC_ROUTES = ['/', '/login', '/register', '/forgot']

router.beforeEach(async (to) => {
  const token = localStorage.getItem(STORAGE_KEYS.ACCESS_TOKEN)
  const isPublicRoute = PUBLIC_ROUTES.includes(to.path)

  // ━━━ 场景1：访问公开页面 ━━━
  if (isPublicRoute) {
    // 已登录用户不应再看到登录页，自动跳到首页
    if (token && to.path === '/login') {
      return '/home/index'
    }
    return true  // 放行
  }

  // ━━━ 场景2：访问需要登录的页面，但没有 token ━━━
  if (!token) {
    return '/login'  // 拦截，跳转登录页
  }

  // ━━━ 场景3：有 token，确保用户信息已加载 ━━━
  const userStore = useUserStore()
  if (!userStore.userInfo) {
    try {
      await userStore.fetchUserBasicInfo()
    } catch {
      // 用户信息获取失败（token 可能已过期）
      userStore.clearAll()
      return '/login'
    }
  }

  return true  // 一切正常，放行
})
```

```

        │
        ▼
   是公开页面？ ──是──→ 已登录访问登录页？ ──是──→ 跳转 /home/index
        │                    │

```

### 4.6 守卫与 http.ts 拦截器的分工

两者不是替代关系，而是**互补**：

| 层级 | 职责 | 时机 |
|------|------|------|
| **路由守卫** | 页面级别的访问控制 | 页面跳转**之前** |
| **http.ts 拦截器** | 请求级别的 Token 管理 | API 请求发出/返回时 |

```
路由守卫：你有门禁卡吗？没有不让进大楼。
http 拦截器：你进了大楼，刷卡开门时发现卡过期了，帮你自动续期。
```

两者配合后的完整认证流程：

```
1. 用户访问 /home/score
2. 路由守卫检查：有 token → 放行
3. 页面渲染，发起 API 请求
4. http.ts 请求拦截器：自动带上 Authorization 头
5. 后端返回 403（token 过期）
6. http.ts 响应拦截器：自动刷新 token，重试请求
7. 刷新成功 → 用户无感知
8. 刷新失败 → 清除 token，跳转登录页
```

## 八、总结

| 问题 | 方案 |
|------|------|
| 通用代码散落各处 | 统一收入 `src/common/`，配置 `@common` 别名 |
| 配置文件太多（16 个） | 精简到 11 个：Tailwind v4 删 3 个，去测试删 2 个 |
| 配置文件能否移入子目录 | **不能**（工具链限制），用 VSCode 文件嵌套替代 |
| `.editorconfig` 有 ESLint/Prettier 后还需要吗 | Prettier 已完全覆盖，个人项目可删除 |
| `tailwind.config.js` + `postcss.config.cjs` | 升级 Tailwind v4，两个文件都不再需要 |
| `tsconfig` 为什么要 3-4 个 | 浏览器/Node 环境不同，需要隔离类型；2-3 个是 Vue 生态标准 |
| 路由守卫有什么用 | 页面级访问控制，替代当前"先渲染再报错"的体验问题 |
| 路由守卫和 http 拦截器的关系 | 互补：守卫管"能不能进页面"，拦截器管"请求的 token 处理" |
