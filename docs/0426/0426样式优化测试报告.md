# 保研助手平台 0426 样式优化测试报告

> 测试范围：
> - 学生端：`idfrontend`
> - 教师端 / 后台管理端：`idfrontend-admin`
>
> 本次改动遵循 `0426样式优化文档.md`：只优化样式、布局适配和展示性能相关细节，不改变业务接口、表单字段、审核逻辑、上传逻辑和 AI 对话流程。

## 一、本次优化内容

### 1.1 学生端 `idfrontend`

主要优化点：

1. **统一全局视觉基础**
   - 调整 `src/assets/main.css` 中的主题色，主色从偏紫改为更清晰的蓝色系。
   - 增加 `page-surface`、`page-toolbar`、`responsive-grid`、`table-scroll` 等通用样式类。
   - 统一 Element Plus 卡片、按钮、表格、输入框、弹窗和移动端表单表现。

2. **优化页面容器**
   - `src/App.vue` 中的 `.page-container` 和 `.page-container-card` 增加最大宽度和居中布局。
   - 让电脑端页面不空散，移动端保持紧凑。

3. **弱化大背景图对视觉和性能的影响**
   - `src/views/home/homelayout.vue` 中降低背景图可见强度。
   - 使用更浅的背景遮罩，让页面主体更清爽。
   - 恢复主内容区域可见滚动条，提升可用性。

4. **优化顶部导航**
   - `src/views/home/component/TopBar.vue` 使用更轻的白色半透明导航栏。
   - 菜单激活色统一到蓝色系。
   - 头像地址复用 Pinia store 中的 `avatarUrl`，避免组件内重复请求头像预览接口。

5. **优化首页视觉**
   - `src/views/home/index.vue` 将深色玻璃风格改为白色卡片风格。
   - 功能卡片改为浅色、轻边框、轻阴影，整体更明朗。
   - 电脑 / 平板 / 手机三端卡片布局保持自适应。

6. **优化 AI 助手响应式**
   - `src/views/ai-agent/index.vue` 修正非标准 Tailwind 尺寸类。
   - 对话框宽高改为 `min()` 自适应，避免手机端超出屏幕。
   - 对话框位置计算增加移动端边界保护。

### 1.2 教师端 `idfrontend-admin`

主要优化点：

1. **统一管理端全局样式**
   - `src/assets/main.css` 补充管理端文字色、辅助色、容器样式。
   - 增加 `admin-section`、`table-scroll`、`responsive-grid` 等通用样式。
   - 优化 `admin-toolbar`，让筛选区成为统一的白色工具栏。
   - 表格表头加粗，提升管理系统阅读效率。

2. **优化后台布局**
   - `src/views/home/component/show.vue` 移除全局 `user-select: none`，避免无法复制表格内容、ID、错误信息。
   - 主内容区增加细滚动条，适合管理后台长表格页面。

3. **优化侧边栏适配**
   - `src/views/home/component/SideMenu.vue` 在窗口恢复到大屏时自动展开，小屏自动折叠。
   - 侧边栏菜单增加圆角、选中态和轻边框。
   - 手机端侧边栏使用 fixed 定位和阴影，减少对内容区的挤压。

4. **优化顶部导航**
   - `src/views/home/component/TopNave.vue` 改为白色半透明顶部栏。
   - 移除无实际生效的 `dark:` 展示依赖。
   - 退出登录时调用 `userStore.clearAll()`，清理内存中的用户状态和角色状态。

5. **优化后台首页**
   - `src/views/home/index.vue` 给定时器增加 `onUnmounted` 清理，避免页面离开后定时器继续运行。

6. **优化 AI 助手响应式**
   - `src/views/ai-agent/index.vue` 对话框宽高改为视口自适应。
   - 对话框位置计算增加手机端边界保护。

7. **修复可见文案问题**
   - `src/stores/profile.ts` 中被编码破坏的错误提示文案恢复为中文。

## 二、Pinia Store 检查结果

### 2.1 学生端

结论：**需要保留 Pinia**。

原因：

- `src/stores/user.ts` 保存用户信息、学生信息、头像地址、登录状态。
- 路由守卫和顶部导航都需要跨页面共享用户状态。
- 头像预览地址带有效期，放在 store 中缓存更合理。

本次优化点：

- 顶部导航不再自己调用头像接口，改为复用 store 中的 `avatarUrl`。
- 减少重复请求，也让头像状态来源更清晰。

后续建议：

- 新代码统一从 `stores/user` 引入。
- `stores/profile.ts` 可以保留作为兼容层，但不要继续扩展。

### 2.2 教师端

结论：**需要保留 Pinia**。

原因：

- `src/stores/profile.ts` 保存当前用户信息和角色列表。
- 路由守卫和菜单过滤依赖角色状态。
- 教师端权限控制比学生端更依赖全局状态。

本次优化点：

- 退出登录时调用 `userStore.clearAll()`，避免只删 localStorage 而内存状态仍残留。

后续建议：

- 只把用户、角色、权限这类跨页面状态放进 Pinia。
- 表格页码、搜索条件、弹窗状态、临时表单状态继续放在页面组件内。

## 四、当前仍需注意的问题

### 4.1 学生端类型检查失败的主要原因

这些问题与样式优化无直接关系，主要是 DTO / Store 类型和页面使用字段不一致：

- `contact` 页面使用了 `studentId`、`enrollmentYear`、`calculatedScore`、`ruleValues` 等类型中不存在的字段。
- `demand/manage.vue` 使用了 `email`、`code`、`studentEmail`、`foreignLanguageLevel` 等类型中不存在的字段。
- `files/index.vue` 中部分变量被推断为 `never`。
- `profile/index.vue` 中存在空值访问风险。
- `login/index.vue` 中 token 字段可能为 `undefined`。

建议后续单独做一轮“类型与接口契约对齐”。

### 4.2 教师端类型检查失败的主要原因

主要问题：

- `apiEvaluation.ts` 缺少 `EvaluationApplication` 类型定义或导入。
- `account-manage/role.vue` 使用可能为 `undefined` 的值作为索引。
- `score/index.vue` 中 `selectedRecord` 可能为空。
- `FileUtil.vue` 与 `fileUtil.vue` 大小写导入不一致，Windows 可运行，但 Linux / 部署环境可能出问题。
- `scoreAttribute.vue` 和 `scoreTemplate.vue` 中表单扩展字段与接口类型不一致。

建议优先修复 `FileUtil.vue` 大小写问题，因为这类问题在部署到 Linux 时风险较高。

### 4.3 环境问题

当前本地 Node 版本：

```text
v20.0.0
```

项目要求：

```text
^20.19.0 || >=22.12.0
```

建议升级 Node 到 `20.19+` 或 `22.12+`，避免 Vite / pnpm / 依赖包出现兼容警告。

## 五、后续可继续优化的方向

### 5.1 性能方向

1. **压缩图片**
   - 学生端 `bg2` 约 `5.60 MB`，建议转 WebP / AVIF。
   - 登录页或首页可以按移动端和桌面端分别提供不同尺寸。

2. **压缩字体**
   - `ZNtitle.ttf` 约 `2.13 MB`。
   - 可以做字体子集化，只保留平台标题需要的字符。

3. **Element Plus 按需导入**
   - 当前两端都使用全量 Element Plus。
   - 后续可使用 `unplugin-auto-import` 和 `unplugin-vue-components`。

4. **路由级懒加载**
   - 教师端主 JS 约 `1.70 MB`，建议将审核、模板、AI 管理、文件预览等大页面拆成异步 chunk。

5. **AI 助手懒加载**
   - 当前 AI 助手挂在登录后的布局内。
   - 后续可改为用户首次点击悬浮按钮时再加载完整聊天逻辑。

### 5.2 工程质量方向

1. 统一组件文件大小写，例如 `fileUtil.vue`。
2. 修复 DTO 类型与后端返回字段不一致问题。
3. 移除生产环境 `console.log`。
4. 清理教师端未使用依赖，例如确认 `vue-i18n`、`@kjgl77/datav-vue3` 是否仍需要。
5. 为核心页面补充基础视觉回归截图或手动验收清单。

## 六、你可以学习的地方

### 6.1 设计系统思路

这次优化不是逐个页面乱调，而是先统一：

- 色彩
- 间距
- 圆角
- 阴影
- 表格
- 表单
- 弹窗
- 响应式断点

这是前端工程中很重要的能力：**把样式从页面级调整提升到系统级治理**。

### 6.2 响应式布局思路

建议重点理解：

- `clamp()`：让字号、间距在不同屏幕下自动变化。
- `min()` / `max()`：限制弹窗、AI 面板不超出视口。
- `CSS Grid auto-fit/minmax`：做卡片自适应网格。
- `100dvh`：解决移动端浏览器地址栏导致的高度问题。

### 6.3 性能优先的样式策略

面试或项目中可以强调：

- 优先使用 CSS media query，而不是大量 JS resize 监听。
- 动画优先使用 `opacity` 和 `transform`。
- 避免大面积循环动画。
- 大图、字体、UI 库才是前端首屏性能的大头。

### 6.4 状态管理边界

Pinia 不应该什么都放。

适合放 Pinia：

- 用户信息
- 登录状态
- 角色权限
- 跨页面共享状态

不适合放 Pinia：

- 当前页码
- 搜索框输入值
- 弹窗开关
- 临时表单数据

这体现的是状态管理边界意识。

## 七、可以作为面试加分项的优化点

### 7.1 设计系统落地

可以描述为：

> 我将原本分散在页面中的样式抽象为全局设计 token 和通用页面类，统一了卡片、按钮、表格、弹窗、工具栏的视觉规范，降低了后续页面维护成本。

### 7.2 三端响应式适配

可以描述为：

> 我针对手机、平板和电脑设置了不同布局策略：移动端单列和安全边距，平板端折叠菜单，桌面端保持信息密度和多列布局。

### 7.3 性能与视觉平衡

可以描述为：

> 我没有盲目增加动效，而是弱化大背景图、减少复杂动画、使用 CSS 原生能力完成自适应，并通过构建结果定位到大图、字体、UI 库是后续性能优化重点。

### 7.4 Store 使用边界

可以描述为：

> 我检查了 Pinia 的使用场景，保留用户、角色、权限等跨页面状态，同时避免将页面临时状态放入全局 store，减少状态污染。

### 7.5 构建验证意识

可以描述为：

> 样式优化完成后，我分别执行了类型检查和生产构建，区分了本次改动引入的问题与项目既有类型债务，并输出了可跟踪的测试报告。
