# 保研助手平台 0426 样式优化方案

> 适用范围：
> - 学生端：`idfrontend`
> - 教师端 / 后台管理端：`idfrontend-admin`
>
> 本文档只讨论样式、交互呈现、响应式和前端展示性能优化，不修改任何现有业务功能、接口字段、路由语义和数据流程。

---

## 一、优化目标

### 1.1 总体风格

平台当前功能模块已经比较完整，但视觉上存在几个共同问题：

- 页面风格不够统一，不同模块之间的卡片、按钮、表格、表单间距差异较大。
- 学生端偏展示型，教师端偏管理型，但两端没有形成清晰的视觉分工。
- 部分页面依赖大背景图、复杂渐变或局部动画，视觉上有表现力，但对首屏性能和低端设备不够友好。
- 手机、平板、电脑三种尺寸下的布局策略还不够系统，尤其是宽表格、侧边栏、浮动 AI 助手、弹窗表单等组件。

本次样式优化目标是：

1. **简约又明朗**：减少杂乱装饰，建立统一的颜色、间距、圆角、阴影和字体层级。
2. **性能优先**：在视觉效果和性能之间，优先选择轻量 CSS、静态层次、少动画、少大图的方案。
3. **三端适配**：分别覆盖手机、平板、电脑三类视口。
4. **不改功能**：只优化呈现层，不改变表单字段、接口调用、权限逻辑、审核流程、上传流程、AI 对话逻辑。
5. **检查 Pinia 必要性**：确认 store 是否保留、哪些状态适合放 store、哪些不应放 store。

## 二、当前工程样式现状

### 2.1 学生端 `idfrontend`

学生端面向学生，核心页面包括：

- 登录 / 注册 / 找回密码
- 首页
- 加分申请与审核记录
- 文件管理
- 个人信息
- 联系 / 学院信息
- AI 助手

当前工程特点：

- 使用 `Vue 3 + Vite + TypeScript + Element Plus + Tailwind CSS v4 + Pinia`。
- `src/assets/main.css` 已经使用 Tailwind v4 的 `@import "tailwindcss"` 和 `@theme`。
- `App.vue` 中已有部分全局页面容器样式，如 `.page-container`、`.page-title`。
- 学生端已有 `src/common/`，包括通用 HTTP、类型、常量和部分 composable。
- 主布局位于 `src/views/home/homelayout.vue`，顶部导航在 `src/views/home/component/TopBar.vue`。
- 全局 AI 助手组件挂在主布局内，会出现在学生端登录后的页面。
- Pinia 主要集中在 `src/stores/user.ts`，`src/stores/profile.ts` 是兼容导出。

样式问题：

- 背景图和遮罩在多个页面中存在，视觉统一但可能影响首屏加载。
- 部分页面文件较大，例如 `score/index.vue`，样式、状态和业务逻辑集中，不利于后续维护。
- 主内容滚动条被隐藏，虽然视觉干净，但用户不容易判断页面是否还能滚动。
- 部分 Tailwind 类可能不规范，例如特殊尺寸类需要确认是否被 Tailwind v4 正确生成。
- 菜单图标复用度高，模块辨识度不足。
- `/home/demand` 相关入口需要后续单独核对路由是否完整，但本文档不处理功能问题。

### 2.2 教师端 `idfrontend-admin`

教师端面向老师和管理员，核心页面包括：

- 首页
- 文件管理
- 审核管理
- 模板管理
- 学生管理
- 账户 / 角色 / 权限管理
- 系统设置
- AI 配置 / 知识库
- AI 助手

当前工程特点：

- 使用 `Vue 3 + Vite + TypeScript + Element Plus + Tailwind CSS v4 + Pinia`。
- `src/assets/main.css` 已包含全局主题变量、Element Plus 覆盖样式、响应式断点。
- API 已按 `src/api/modules/` 组织。
- 通用代码在 `src/common/`，包括 `http.ts`、`storage.ts`、通用类型。
- 主布局由 `show.vue`、`SideMenu.vue`、`TopNave.vue`、`titleUI.vue` 组成。
- Pinia store 位于 `src/stores/profile.ts`，主要用于用户信息和角色权限。

样式问题：

- 教师端是管理系统，表格、筛选栏、弹窗、批量操作较多，需要比学生端更强调信息密度和可读性。
- 当前侧边栏在小屏只折叠，没有完整的移动端抽屉策略，手机宽度下仍可能拥挤。
- `user-select: none` 如果作用在整个布局，会影响复制表格内容、用户 ID、错误信息等。
- 存在 `dark:` 样式，但如果没有根节点 `.dark` 切换逻辑，暗色样式实际不会生效。
- 全局 AI 助手始终挂载，若组件较重，可考虑后续懒加载或延迟加载。
- Element Plus 全量注册简单稳定，但包体偏大，后续可评估按需导入。

## 三、统一设计原则

### 3.1 视觉关键词

建议统一为：

- **清爽**：背景以浅灰、白色、低饱和蓝为主。
- **明朗**：主按钮、状态标签、重点数字使用明确色彩。
- **克制**：减少强渐变、大阴影、大面积背景图。
- **可信**：教师端偏稳重，学生端偏亲和，但共用同一套基础设计语言。

### 3.2 色彩策略

建议建立两端共享的基础色：

| 用途 | 建议 |
|---|---|
| 主色 | 蓝色系，负责按钮、链接、当前菜单、重点操作 |
| 辅助色 | 青色 / 紫色少量用于学生端模块卡片 |
| 背景色 | `#f5f7fb` 或接近的浅灰蓝 |
| 卡片色 | 白色，保留轻微边框 |
| 文字主色 | 深灰黑，保证可读性 |
| 文字次色 | 中灰，用于说明文字 |
| 成功 / 警告 / 错误 | 优先复用 Element Plus 语义色 |

不建议大量使用高饱和渐变作为主背景。渐变可以保留在：

- 登录页局部装饰
- 首页欢迎区域
- 卡片 hover 的轻微边框或背景

### 3.3 间距和圆角

建议统一：

| 元素 | 建议 |
|---|---|
| 页面外边距 | 电脑 `24px`，平板 `20px`，手机 `12px - 16px` |
| 卡片内边距 | 电脑 `20px - 24px`，手机 `14px - 16px` |
| 组件间距 | `12px / 16px / 24px` 三档 |
| 圆角 | 卡片 `12px`，按钮 `8px`，弹窗 `14px` |
| 阴影 | 默认只用轻阴影，hover 可略增强 |

### 3.4 字体层级

建议统一：

| 层级 | 电脑 | 平板 | 手机 |
|---|---:|---:|---:|
| 页面标题 | 24px - 28px | 22px - 24px | 20px - 22px |
| 区块标题 | 18px - 20px | 17px - 18px | 16px - 17px |
| 正文 | 14px - 16px | 14px | 14px |
| 表格 | 14px | 13px - 14px | 12px - 13px |
| 辅助说明 | 12px - 13px | 12px | 12px |

## 四、学生端样式优化思路

### 4.1 学生端定位

学生端更像一个“个人服务台”，样式应更轻、更亲和：

- 首页强调“我能做什么”
- 加分申请强调流程清晰
- 文件管理强调文件状态和操作入口
- 个人信息强调资料完整度
- AI 助手强调可用性，不喧宾夺主

### 4.2 首页优化

建议方向：

- 使用简洁的欢迎卡片，弱化大面积背景图。
- 功能模块卡片统一高度、圆角、图标尺寸和 hover 效果。
- 模块入口增加简短说明，但避免大段文字。
- 在电脑端使用 `3 - 4` 列卡片，在平板端 `2` 列，手机端 `1` 列。

示例布局策略：

```text
电脑：欢迎区 + 4 列功能卡片 + 最近状态
平板：欢迎区 + 2 列功能卡片
手机：欢迎区 + 单列功能卡片 + 底部留白
```

### 4.3 加分申请页面优化

重点不是增加动画，而是提升阅读效率：

- 筛选区统一为紧凑工具栏。
- 表格在电脑端完整展示，在平板端减少次要列，在手机端改为卡片列表或横向滚动。
- 审核状态使用统一标签色。
- 文件预览 / 下载按钮保持固定视觉样式。

### 4.4 文件管理页面优化

建议：

- 上传区保持突出，但不要占据过高首屏。
- 文件列表使用轻卡片或表格两套响应式策略。
- 手机端文件操作按钮改为纵向或更多菜单，避免按钮挤压。
- 文件类型标签使用统一颜色，不要每个页面自定义。

### 4.5 AI 助手优化

建议：

- 默认保持轻量悬浮入口。
- 手机端入口缩小并靠右下角，避免遮挡表单提交按钮。
- 聊天面板使用 `min(宽度, 视口宽度 - 安全边距)`。
- 减少复杂动画，只保留打开 / 收起的简单过渡。
- 后续可考虑组件懒加载，降低首屏压力。

## 五、教师端样式优化思路

### 5.1 教师端定位

教师端是管理系统，样式应更稳、更清晰：

- 信息密度高，但不能拥挤。
- 表格、筛选、批量操作是核心。
- 权限、模板、审核等页面需要减少视觉噪音。
- 重点信息要比装饰更突出。

### 5.2 后台整体布局

建议保留：

- 顶部栏
- 左侧菜单
- 主内容区

但需要进一步优化：

- 电脑端：左侧菜单默认展开，主内容区最大化展示表格。
- 平板端：左侧菜单默认折叠，只显示图标，用户可展开。
- 手机端：左侧菜单建议升级为抽屉式，不占用页面固定宽度。

响应式策略：

```text
>= 1200px：标准后台布局，侧边栏展开
768px - 1199px：侧边栏折叠，内容区优先
< 768px：侧边栏抽屉化，顶部显示菜单按钮
```

### 5.3 表格页面优化

教师端大量页面都是表格，建议统一表格样式：

- 表格卡片外层统一 `admin-page`。
- 筛选栏统一 `admin-toolbar`。
- 操作按钮统一放在右侧或表格行尾。
- 批量操作和危险操作使用明确颜色区分。
- 手机端不强求完整表格，可以使用：
  - 横向滚动表格
  - 卡片列表
  - 只显示关键列，更多信息展开查看

优先推荐：

| 场景 | 方案 |
|---|---|
| 审核列表 | 电脑表格，手机卡片 |
| 学生管理 | 电脑表格，手机横向滚动 |
| 权限管理 | 电脑表格，手机卡片 |
| 模板规则 | 电脑分组表格，手机折叠面板 |

### 5.4 模板配置页面优化

模板相关页面信息层级多，建议：

- 使用分区卡片：基础信息、规则配置、属性配置。
- 表单宽度限制在可读范围内，不要整行铺满。
- 规则项使用可折叠卡片，避免页面过长。
- 必填项、单位、说明文字样式统一。

### 5.5 系统设置和权限页面优化

建议：

- 权限和角色页面减少装饰，突出结构。
- 危险操作按钮统一使用 `danger`。
- 分配权限弹窗使用树形结构或分组卡片时，保持滚动区域固定高度。
- 系统设置使用“设置项卡片”，避免所有内容堆在一个表单里。

## 六、三端适配方案

### 6.1 断点建议

建议统一使用以下断点：

| 设备 | 宽度范围 | 布局目标 |
|---|---|---|
| 手机 | `< 768px` | 单列、抽屉、底部安全距离 |
| 平板 | `768px - 1199px` | 双列、折叠菜单、适中字号 |
| 电脑 | `>= 1200px` | 多列、完整菜单、完整表格 |

### 6.2 页面容器

建议两端统一页面容器：

```css
.page-shell {
  width: min(100%, 1440px);
  margin: 0 auto;
  padding: clamp(12px, 2vw, 24px);
}
```

学生端可以叫 `.page-container`，教师端可以叫 `.admin-page`，但底层 spacing 规则应一致。

### 6.3 卡片网格

建议使用 CSS Grid：

```css
.responsive-grid {
  display: grid;
  grid-template-columns: repeat(auto-fit, minmax(240px, 1fr));
  gap: 16px;
}
```

优势：

- 不需要复杂 JS。
- 手机自动单列。
- 平板自动双列。
- 电脑自动多列。
- 性能优于频繁监听窗口尺寸。

### 6.4 表格适配

Element Plus 表格建议：

- 外层包裹可横向滚动容器。
- 电脑端完整列。
- 平板端降低字号和列宽。
- 手机端优先做卡片化展示；如果改动风险较大，先保留横向滚动。

### 6.5 弹窗适配

建议统一：

```text
电脑：宽度 720px - 960px
平板：宽度 80vw - 90vw
手机：宽度 94vw - 96vw，高度不超过 90dvh，内部滚动
```

## 七、性能优先的样式优化策略

### 7.1 优先使用 CSS，少用 JS

优先级：

1. CSS media query
2. CSS Grid / Flex
3. Tailwind 响应式类
4. 少量 resize 监听
5. 避免复杂 JS 布局计算

### 7.2 动画策略

保留：

- 按钮 hover
- 卡片 hover
- 弹窗进入 / 离开
- AI 助手展开 / 收起

减少：

- 大面积背景动画
- 长时间循环动画
- 多元素同时 transform
- 页面加载时复杂动效

建议所有动画遵循：

```text
持续时间：150ms - 240ms
属性：opacity / transform 优先
避免：width / height / top / left 频繁变化
```

### 7.3 图片策略

建议：

- 背景图只保留在登录页或首页局部。
- 大图使用压缩后的 WebP / AVIF。
- 移动端使用更小尺寸图片。
- 装饰图不要阻塞主要内容展示。

### 7.4 Element Plus 优化

当前两端都全量引入 Element Plus，优点是稳定简单，缺点是包体偏大。

后续可选优化：

- 使用 `unplugin-auto-import` 和 `unplugin-vue-components` 按需引入。
- 样式按需导入。
- 把大型页面按路由懒加载。

但这属于构建层优化，建议放在样式统一完成之后再做，避免一次改动过大。

## 八、Pinia Store 检查

### 8.1 学生端是否需要 Pinia

结论：**需要保留**。

学生端 `src/stores/user.ts` 主要承担：

- 用户登录状态
- 用户基础信息
- token 存在状态
- 头像地址
- 学生信息绑定状态
- 路由守卫中的用户信息加载

这些状态跨页面共享，适合放在 Pinia。

建议：

- 保留 `user.ts` 作为主 store。
- `profile.ts` 如果只是兼容导出，可以继续保留，但后续新代码统一从 `stores/user` 引入。
- 头像加载逻辑建议尽量集中到 store，避免组件和 store 双重请求。

### 8.2 教师端是否需要 Pinia

结论：**需要保留**。

教师端 `src/stores/profile.ts` 主要承担：

- 当前用户信息
- 当前用户角色列表
- 路由权限判断
- 侧边栏菜单过滤
- 登录后用户信息加载

这些状态关系到权限和导航，适合放在 Pinia。

建议：

- 保留教师端 `useUserStore`。
- 退出登录时应统一调用 store 的清理方法，避免内存状态残留。
- 不建议把表格查询参数、弹窗开关、临时表单都放进全局 store，这些应保留在页面组件内部。

### 8.3 哪些状态不建议放 Pinia

不建议放入全局 store：

- 表格当前页码
- 搜索框输入值
- 弹窗开关
- 当前编辑表单
- 临时上传进度
- 某个页面内部的筛选项

除非它们需要跨页面保持，否则放在组件内即可，性能和维护成本更低。

## 九、建议执行阶段

### Phase 1：统一视觉基础

目标：不碰业务功能，只统一全局视觉规则。

建议改动：

- 整理 `main.css` 中的颜色变量、字体、背景、卡片、按钮、表格、弹窗规则。
- 统一 `.page-container` / `.admin-page` / `.admin-toolbar`。
- 统一 Element Plus 的按钮、表格、输入框、弹窗样式。
- 减少过重阴影和大面积渐变。

风险：低。

### Phase 2：布局响应式

目标：保证手机、平板、电脑都能正常查看。

建议改动：

- 学生端首页、文件页、加分页适配手机。
- 教师端侧边栏在手机端改为抽屉式。
- 教师端表格页面统一横向滚动或卡片策略。
- 弹窗统一移动端宽度和内部滚动。

风险：中。

### Phase 3：组件视觉统一

目标：减少重复样式，让页面风格一致。

建议改动：

- 抽取通用卡片样式。
- 抽取状态标签样式。
- 抽取页面工具栏样式。
- 统一文件上传组件和文件列表样式。
- 统一 AI 助手悬浮入口样式。

风险：中。

### Phase 4：性能与细节优化

目标：让页面更轻、更稳定。

建议改动：

- 压缩或替换大背景图。
- 减少循环动画。
- 移除生产环境 console。
- 检查未使用依赖。
- 评估 Element Plus 按需导入。
- AI 助手考虑懒加载。

风险：中，需要构建验证。

## 十、验收标准

### 10.1 功能不变

- 登录、注册、找回密码正常。
- 学生端加分申请、文件管理、个人信息、AI 助手正常。
- 教师端审核、模板、学生、角色权限、系统设置正常。
- 接口字段和请求路径不变。
- 路由权限逻辑不变。

### 10.2 样式一致

- 页面背景、卡片、按钮、表格、弹窗风格统一。
- 学生端和教师端有区分，但属于同一套设计语言。
- 重点状态颜色清晰。
- 危险操作醒目。

### 10.3 三端适配

- 手机端不出现主体内容被侧边栏遮挡。
- 平板端菜单和表格不挤压。
- 电脑端信息密度合适，不空旷。
- 弹窗不超出屏幕。
- 表格可阅读、可操作。

### 10.4 性能可接受

- 首屏不要被大图和复杂动画拖慢。
- 页面切换无明显卡顿。
- AI 助手不影响主页面首屏。
- 不引入新的大型 UI 动效库。

---

## 十一、推荐优先处理清单

优先级从高到低：

1. 统一两端全局主题变量和 Element Plus 覆盖样式。
2. 教师端表格页和筛选栏统一样式。
3. 教师端移动端侧边栏优化。
4. 学生端首页和功能卡片优化。
5. 学生端加分申请 / 文件管理移动端适配。
6. AI 助手悬浮入口和面板响应式优化。
7. 检查大背景图、循环动画、console、未使用依赖。
8. Pinia 只保留跨页面用户和权限状态，页面临时状态不进 store。

---

## 十二、结论

本次样式优化建议采取“先统一设计系统，再逐页适配”的方式，而不是直接逐个页面微调。这样可以保证学生端和教师端风格统一，也能避免重复样式越来越多。

总体建议：

- 学生端：突出亲和、流程清晰、操作轻量。
- 教师端：突出效率、信息密度、表格可读性。
- 两端共享：颜色、间距、圆角、卡片、按钮、表格、弹窗规则。
- 性能策略：少动画、少大图、CSS 优先、组件懒加载按需评估。
- Pinia：两端都需要保留，但只用于用户、权限、跨页面共享状态。

后续如果确认该方案，可以再进入代码实施阶段，按 Phase 1 到 Phase 4 分批修改，每批修改后分别验证学生端和教师端构建与主要页面显示。
